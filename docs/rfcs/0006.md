## Problem

Без чётких правил открытия/сохранения файлов и обработки ошибок редактор теряет надёжность.
Отдельно нужен прозрачный readonly-режим.

## Goal

Зафиксировать поведение file I/O для MVP, включая ошибки доступа и readonly.

## Design

### Открытие файла

- Запуск: `be <path>`.
- Если файл не существует — попытаться создать его автоматически как пустой файл.
- Если создать файл невозможно (нет прав, отсутствует каталог, I/O ошибка) — показать явную ошибку и код выхода `1`.
- При успешном открытии (существующего или только что созданного файла) содержимое загружается в буфер.

### Сохранение

- Сохранение по `Ctrl+S` в исходный путь.
- При ошибке записи показывается сообщение, приложение не завершается.
- После успешного сохранения `modified` сбрасывается в `saved`.

### Readonly

- Флаг: `be --readonly <path>`.
- Редактирование и сохранение блокируются.
- В UI показывается, что документ открыт в readonly-режиме.

### Права доступа

- Если чтение возможно, но записи нет — открыть файл в readonly-поведении для сохранения,
  с явным сообщением при попытке `Ctrl+S`.

## Alternatives

1. **Не создавать файл автоматически (ошибка при отсутствии файла)**
   - Плюсы: более строгая и предсказуемая модель доступа.
   - Минусы: хуже UX для заметок и быстрых черновиков.

2. **Немедленно завершать приложение при ошибке сохранения**
   - Плюсы: простая логика.
   - Минусы: риск потери несохранённых правок в памяти.

## Voting

Предлагаемое решение: принять строгие правила открытия и устойчивое поведение при ошибках сохранения.

Статус голосования: `pending`.

## Migration
