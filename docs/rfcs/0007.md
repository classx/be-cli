## Problem

Нужна единая модель хранения и применения настроек. Без этого настройки из `Ctrl+O`, CLI-флагов
и дефолтов могут конфликтовать.

## Goal

Определить источник конфигурации, формат и правила приоритета настроек.

## Design

### Путь конфигурации

- Основной путь: `$XDG_CONFIG_HOME/be/config.toml`
- Fallback: `~/.config/be/config.toml`

### Автосоздание

- Если файла конфигурации нет, он создаётся автоматически при первом запуске.
- В файл записываются дефолтные значения MVP.

### Параметры MVP

- `lines_before`
- `lines_after`
- `cursor_on_open = "start" | "end"`

### Приоритет источников

1. CLI-флаги текущего запуска
2. Изменения из панели настроек (`Ctrl+O`) в текущей сессии
3. Конфигурационный файл
4. Встроенные дефолты

### Валидация

- `lines_before`/`lines_after` должны быть `>= 0`.
- Некорректные значения заменяются безопасными дефолтами с предупреждением в UI.

## Alternatives

1. **Хранить настройки только в памяти (без файла)**
   - Плюсы: проще MVP.
   - Минусы: нет persistency между запусками.

2. **Использовать JSON/YAML вместо TOML**
   - Плюсы: привычные форматы.
   - Минусы: TOML удобнее для небольшого CLI-конфига и хорошо читается вручную.
