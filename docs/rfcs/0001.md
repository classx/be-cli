## Problem

Для ежедневных заметок я использую Vim, однако на больших экранах сложно удерживать фокус на текущем фрагменте текста.

## Goal

Нужен минималистичный текстовый редактор, который отображает на чистом экране только небольшой фрагмент файла и позволяет его редактировать.
В идеале строка с курсором находится в центре экрана и выделяется ярче, а 2–3 предыдущие и следующие строки отображаются более приглушённо.

## Design

### Scope (MVP)

MVP покрывает один сценарий: открыть существующий текстовый файл, редактировать его в фокусном режиме и сохранить изменения.

Функциональные требования MVP:
- Отображение ограниченного окна контекста: текущая строка + 2–3 строки сверху и снизу.
- Центрирование активной строки по вертикали (если позволяет размер терминала).
- Визуальное выделение активной строки (ярче), соседних строк — приглушённо.
- Базовое редактирование: вставка/удаление символов, `Enter`, `Backspace`, перемещение курсора стрелками.
- Сохранение файла (`Ctrl+S`) и выход (`Ctrl+Q`, при несохранённых изменениях — подтверждение).
- Строка статуса внизу экрана:
  - слева — `[filename] [saved|modified]`;
  - справа — `Ln <line>, Col <col>` (например, `Ln 12, Col 8`).
- Быстрые действия из редактора:
  - `Ctrl+O` — показать настройки;
  - `?` — показать помощь.

### Настройки (`Ctrl+O`)

Настройки MVP:
- `lines_before` — количество строк контекста до активной строки.
- `lines_after` — количество строк контекста после активной строки.
- `cursor_on_open` — стартовая позиция курсора при открытии файла:
  - `start` — начало файла;
  - `end` — конец файла (удобно для заметок).

Файл конфигурации:
- путь по XDG: `$XDG_CONFIG_HOME/be/config.toml`;
- fallback: `~/.config/be/config.toml`.
- если файла конфигурации нет, он создаётся автоматически при первом запуске с настройками по умолчанию.

Опционально (после MVP):
- настройка темы (контраст, яркость активной строки, приглушение контекста);
- настройка горячих клавиш (переназначение базовых действий);
- отдельная горячая клавиша для режима «показать весь файл» (только превью):
  - открывает полный текст файла в режиме просмотра;
  - поддерживает скроллинг;
  - выход из режима возвращает пользователя в фокусный режим на прежнюю позицию курсора.

### Non-goals (для MVP)

Не входят в первую версию:
- Многобуферность и вкладки.
- Плагины и расширяемость.
- Сложные режимы редактирования (модальность в стиле Vim, макросы, multi-cursor).
- Встроенный файловый менеджер.

### CLI-интерфейс

Предлагаемый минимальный интерфейс запуска:

- `be <path>` — открыть файл.
- `be --lines 3 <path>` — число видимых строк контекста сверху/снизу.
- `be --readonly <path>` — просмотр без записи.

Ошибки:
- Если файл не существует, попытаться создать его автоматически.
- Если создать файл невозможно (например, нет прав или каталога), показать явную ошибку и код выхода `1`.
- Если нет прав на запись, разрешить чтение, но блокировать сохранение с понятным сообщением.

### Архитектура (высокоуровнево)

Компоненты:
- `Input` — обработка клавиш терминала.
- `Buffer` — текстовый буфер и позиция курсора.
- `Viewport` — расчёт видимого окна (центрирование + контекст).
- `Renderer` — отрисовка в терминал (стили активной/неактивных строк).
- `File I/O` — загрузка и сохранение.

Поток данных:
`Input -> Buffer -> Viewport -> Renderer`.

### Язык реализации

Текущий приоритет для реализации — `Hare`.

Это **не окончательный выбор**: при необходимости (например, из-за ограничений экосистемы, сложности реализации TUI/ввода или требований к portability) допускается возврат к `Rust`.

Критерии пересмотра выбора языка:
- сложность реализации и поддержки терминального UI;
- доступность и зрелость необходимых библиотек;
- скорость итераций и удобство сопровождения проекта.

### Критерии готовности MVP

- Редактирование и сохранение файла работает стабильно на Linux-терминалах (xterm/kitty/alacritty/gnome-terminal).
- На экране всегда видна активная строка и заданный контекст.
- Нет мерцания интерфейса при обычной печати и перемещении курсора.

## Alternatives

### A. Использовать готовый инструмент без разработки

Готовые Linux CLI-альтернативы:

1. **Vim/Neovim + плагины фокуса** (`goyo.vim`, `limelight.vim`, `zen-mode.nvim`)
   - Плюсы: мощно, гибко, уже зрелая экосистема.
   - Минусы: настройка и плагины добавляют сложность; продукт остаётся «общим редактором», а не специально фокусным инструментом.

2. **Helix**
   - Плюсы: современный TUI-редактор, хорошая базовая ergonomics, простая установка.
   - Минусы: ориентирован на универсальное код-редактирование, а не на предельно минималистичный режим заметок.

3. **Kakoune**
   - Плюсы: лёгкий и очень клавиатурно-ориентированный редактор.
   - Минусы: порог входа и модель редактирования специфичны; не решает напрямую задачу «чистого фокусного экрана».

4. **micro**
   - Плюсы: простой старт, дружелюбный интерфейс в терминале.
   - Минусы: по умолчанию не даёт нужного узкого фокусного представления с центрированием активной строки.

5. **nano / joe / mg**
   - Плюсы: доступны почти везде, минимальные зависимости.
   - Минусы: ограниченная настраиваемость и отсутствие целевого UX для фокусного редактирования.

6. **WordGrinder**
   - Плюсы: ориентирован на набор текста в терминале, мало отвлекающих элементов.
   - Минусы: менее универсален как редактор файлов общего назначения; экосистема и привычные workflow ограничены.

### B. Обёртка вокруг существующего редактора

Идея: запускать внешний редактор с преднастроенным «focus mode».
- Плюсы: очень быстрый старт разработки.
- Минусы: высокая зависимость от внешних инструментов, нестабильность UX между окружениями, сложнее поддерживать предсказуемое поведение.

### C. Разработать отдельный узкоспециализированный CLI-инструмент (предпочтительно)

- Плюсы: полный контроль над UX, минимальный и предсказуемый интерфейс, отсутствие лишних возможностей.
- Минусы: стоимость разработки и поддержки.

**Вывод:** для цели RFC предпочтителен вариант C (собственный минималистичный редактор), при этом альтернативы из пункта A остаются рабочими обходными решениями «уже сегодня».

## Voting

Предлагаемое решение: **принять RFC и реализовать MVP из раздела Design**.

Критерии принятия:
- Подтверждённая ценность фокусного режима для ежедневных заметок.
- Согласие, что существующие инструменты не закрывают задачу в нужной степени простоты.
- Готовность поддерживать узкоспециализированный CLI-инструмент.

Статус голосования: `pending`.

## Migration

Миграция как таковая не требуется: это новый инструмент, не заменяющий автоматически существующие редакторы.

План внедрения:
1. Реализовать MVP-функциональность.
2. Использовать инструмент параллельно с текущим процессом заметок.
3. Оценить удобство по итогам 1–2 недель использования.
4. При положительном результате — обновить статус RFC на `accepted` и зафиксировать roadmap следующей версии.
